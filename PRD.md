# Mianotes Web Service PRD

## Overview

Mianotes turns files, links, images, audio, and text into organised, actionable Markdown notes using AI-powered parsing and summarisation tools.

The web service app is the backend API for Mianotes. It is designed to run primarily on a local server, store note content on the filesystem, keep lightweight indexes in SQLite, and expose JSON REST APIs for the React web app and external developers.

## Product Goals

- Convert user-provided content into clean Markdown notes.
- Store notes and source files in a transparent filesystem layout.
- Keep an indexed record of users, topics, notes, source files, and comments.
- Support API access through per-user tokens.
- Start with SQLite while keeping the storage layer adaptable for PostgreSQL later.
- Use OpenAI ChatGPT APIs for note generation in v1.
- Use LiteParse for supported document and image parsing in v1.
- Keep the system simple enough for local self-hosting, but structured enough to scale.

## Non-Goals For V1

- Full multi-database support at runtime.
- YouTube transcript extraction.
- Local LLM note generation.
- Complex tagging or multi-topic notes.
- Relocating saved notes after creation.

## Users And Authentication

Users register with:

- Valid email address
- Display name

Email is the unique user identifier.

The app generates:

- A stable internal user ID
- An autogenerated username derived from the email address using a hash-like format
- A unique API token per user

Each user token is used for developer/API authentication. The React web app also needs an authenticated way to call the same APIs. The implementation should support browser-friendly auth separately from raw developer tokens if needed.

Token rules:

- Tokens are generated by the app.
- Tokens are stored on the filesystem.
- Each user has their own token.
- If a user token is deleted, the app should be able to generate a new random token.
- API responses should never expose raw token values unless the endpoint is explicitly designed for token creation/regeneration.

## Storage Model

Mianotes uses the filesystem as the main storage layer for user-facing content.

Primary content storage:

- Markdown files for generated notes
- Original source files for uploads
- JSON sidecar files for note comments

Index storage:

- SQLite in v1
- Repository/adapter layer designed so PostgreSQL can be supported later

### Filesystem Layout

Generated notes are stored under:

```text
/data/<username>/<topic>/<filename>.md
```

Source files are stored in the same folder using the same base filename and their original extension:

```text
/data/<username>/<topic>/<filename>.<original-extension>
```

Comments are stored as JSON metadata sidecars:

```text
/data/<username>/<topic>/<filename>.comments.json
```

Once a Markdown note path is created, it must not change. The database stores the note path permanently to avoid maintenance overhead from moves and renames.

### Plain Text Inputs

For plain text note creation, v1 should save a source text file alongside the generated note when it is useful for auditability and reprocessing:

```text
/data/<username>/<topic>/<filename>.source.txt
```

This keeps all note types consistent: every generated note can have a traceable source file when practical.

## Database Model

The database keeps indexes and relationships for fast lookup, sorting, and API responses. The source of truth for note body text remains the Markdown file.

### users

Supports:

- Insert
- Update
- Delete

Fields:

- id
- email
- name
- username
- created_at
- updated_at

Email must be unique. Username is autogenerated from email.

### topics

Supports:

- Insert
- Update
- Delete

Fields:

- id
- user_id
- name
- slug
- created_at
- updated_at

Topics belong to users. One note belongs to exactly one topic.

### notes

Supports:

- Insert
- Update title
- Update text
- Delete

Fields:

- id
- user_id
- topic_id
- title
- note_path
- created_at
- updated_at

Notes have a relationship with `users` and `topics`.

Notes must include timestamps so they can be sorted in API responses and in the web app.

The API should return `text` by reading the Markdown file from `note_path`.

### source_files

Supports:

- Insert
- Delete

Fields:

- id
- note_id
- file_path
- original_filename
- content_type
- created_at

Source files belong to notes. The API response for a note should include source file information and full URL paths for accessing source files.

### comments

Comments are metadata for a note and are stored as JSON files on disk.

Database fields:

- id
- note_id
- comments_path
- created_at
- updated_at

The JSON sidecar file contains the actual comment data.

### Link Inputs

For v1, links should be fetched and converted into clean textual content when possible, then sent to OpenAI for note generation.

The original idea was to convert links to PDFs and process them through LiteParse. That may be useful later for uniformity, but v1 should avoid unnecessary PDF generation if direct text extraction gives reliable results.

The app should preserve the original URL as source metadata.

### YouTube Links

YouTube transcript extraction is planned but not required for v1.

Future behavior:

- If a transcript is available, use the transcript directly.
- If no transcript is available, fall back to another supported ingestion method.

### Audio Inputs

Audio parsing is not finalized.

OpenAI should be evaluated for transcription support. V1 architecture should leave room for an audio transcription step before Markdown generation.

## Note Creation Pipeline

Every note creation flow follows this high-level pipeline:

1. Accept source input: upload, link, or plain text.
2. Store the source file or source representation on disk.
3. Parse source content.
4. Send parsed content to OpenAI ChatGPT API.
5. Generate a Markdown note.
6. Generate or infer a note title.
7. Save the Markdown file under `/data/<username>/<topic>/<filename>.md`.
8. Save comments sidecar as `/data/<username>/<topic>/<filename>.comments.json`.
9. Insert database records for the note and source file.
10. Return a JSON representation of the note.

## AI Requirements

V1 uses OpenAI ChatGPT API for Markdown note generation.

The generated note should be:

- Organised
- Actionable
- Useful as a long-term note
- Valid Markdown

Expected output sections may include:

- Summary
- Key points
- Action items
- Decisions
- Questions
- References

The exact prompt contract should be defined during implementation.

## Title Generation

Each note has a title.

The API creates an initial title automatically using one of these strategies:

- Use the first meaningful 200 characters of the generated note.
- Ask a small title-generation model to propose a title based on the first 1000 characters.

For v1, OpenAI can perform title generation to avoid introducing a second model runtime immediately.

Local small-model support, such as Ollama, should be researched separately. The current hardware target may have only 4 GB RAM, so model choice must be conservative.

## API Requirements

All APIs return JSON.

The API follows REST conventions.

Every returned note should include:

- Note ID
- User ID
- User details needed by the frontend
- Topic ID
- Topic details needed by the frontend
- Created timestamp
- Updated timestamp
- Title
- Markdown text
- Full URL to the Markdown file
- Source file metadata
- Full URL to source files
- Comments metadata
- API action metadata showing URLs and methods available for the note

### API Action Metadata

Each note response should include API action hints so the frontend can show developers how to call the API.

Example shape:

```json
{
  "actions": {
    "self": {"method": "GET", "url": "/api/notes/{note_id}"},
    "update": {"method": "PATCH", "url": "/api/notes/{note_id}"},
    "delete": {"method": "DELETE", "url": "/api/notes/{note_id}"},
    "comments": {"method": "GET", "url": "/api/notes/{note_id}/comments"}
  }
}
```

## Candidate REST API

### Users

- `POST /api/users`
- `GET /api/users/{user_id}`
- `PATCH /api/users/{user_id}`
- `DELETE /api/users/{user_id}`
- `POST /api/users/{user_id}/tokens`

### Topics

- `POST /api/topics`
- `GET /api/topics`
- `GET /api/topics/{topic_id}`
- `PATCH /api/topics/{topic_id}`
- `DELETE /api/topics/{topic_id}`

### Notes

- `POST /api/notes`
- `GET /api/notes`
- `GET /api/notes/{note_id}`
- `PATCH /api/notes/{note_id}`
- `DELETE /api/notes/{note_id}`

Note creation should support multiple input types, either via one endpoint with an `input_type` field or separate endpoints.

Candidate specialised endpoints:

- `POST /api/notes/from-file`
- `POST /api/notes/from-link`
- `POST /api/notes/from-text`
- `POST /api/notes/from-audio`

### Source Files

- `GET /api/notes/{note_id}/source-files`
- `GET /api/source-files/{source_file_id}`

### Comments

- `GET /api/notes/{note_id}/comments`
- `POST /api/notes/{note_id}/comments`
- `PATCH /api/notes/{note_id}/comments/{comment_id}`
- `DELETE /api/notes/{note_id}/comments/{comment_id}`

Comment endpoints read and write the note sidecar JSON file.

## Web App API Access

The React web app needs a way to call the APIs.

V1 should distinguish between:

- Browser app authentication for normal users
- Per-user API tokens for developers and integrations

This can be implemented with the same user model, but raw API tokens should not be casually exposed in normal browser responses.

## Performance Requirements

Although Mianotes is designed for local servers, performance should be considered from the start.

V1 should:

- Keep database queries indexed around user, topic, note, and timestamps.
- Avoid scanning the filesystem for normal list views.
- Read Markdown files only when full note content is needed.
- Store stable file paths in the database.
- Use background jobs for long-running parsing and AI generation if synchronous requests become too slow.
- Keep a repository/adapter boundary so SQLite can be replaced or supplemented by PostgreSQL later.

## Initial Technical Direction

Recommended v1 architecture:

- Python API service with a clean app/domain/repository structure.
- SQLite as the default database.
- Filesystem storage service for notes, source files, tokens, and comment JSON.
- OpenAI integration service for note generation.
- LiteParse integration service for document and image parsing.
- Link ingestion service for URL fetching and text extraction.
- Repository interfaces that can later support PostgreSQL.
